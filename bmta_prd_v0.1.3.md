# [PRD] 모든 타는 곳의 이야기 - 브타 (BMTA)

버전: v0.1.2  
작성일: 2026. 01. 30

---

## 1. 제품 개요 (Product Overview)

### 1.1 제품 비전

"대한민국 대중교통 이용객의 지루한 이동 시간을 가치 있는 소통과 정보 공유의 시간으로 바꾼다."

---

### 1.2 제작 배경

대중교통을 이용하는 고객이 각자의 커뮤니티나 SNS에서 소통하는 주제를 하나로 통합

누구나 대중교통에서 발생되는 이슈나 사건 사고 그리고 필요한 내용을 쉽고 빠르게 찾기 위함

향후 지하철의 지하상가 시장 활성화와 스마트 오더 도입으로 간편한 주문

궁극적으로 지하철과 버스 그리고 택시를 이용하는 모든 사람이 모든 것을 확인하고 이용하는 공간

---

### 1.3 목표 (Goals)

버스, 지하철, 택시 이용객을 위한 통합 익명 커뮤니티 구축.

실시간 교통 정보(Utility)와 감성 소통(Community)의 유기적 결합.

실시간 버스 위치와 지하철 위치를 확인하여 사용자들의 대중 교통 이용의 편리함 제공.

앱 포인트 적립 및 결제를 통한 내부 커뮤니티 활성화 형성.

---

### 1.4 타겟 고객 (Target Audience)

직장인/대학생: 매일 정해진 노선으로 출퇴근/통학하며 실시간 정보와 소통을 원하는 층.

택시 이용객: 늦은 밤 안심 귀가 정보 공유나 기사님 관련 리뷰를 남기고 싶은 층.

분실물 발생자/습득자: 빠르고 정확한 노선 기반의 유실물 회수를 원하는 층.

---

## 2. 사용자 경험 및 디자인 (UX & Design)

### 2.1 핵심 디자인 원칙

심리스(Seamless): 정보 확인에서 소통 참여까지 단절 없는 흐름 제공.

익명성(Anonymity): 사용자 신원 보호를 위한 랜덤 닉네임 시스템.

직관성(Intuitive): 이동 중(흔들림 등)에도 조작하기 쉬운 큰 버튼과 간결한 UI.

---

### 2.2 주요 사용자 흐름 (User Flow)

탐색: 노선 검색 또는 노선도에서 역 선택 → 실시간 도착 정보 확인.

참여: 정보 카드 우측 '피드 바로가기' 클릭 → 해당 노선/역의 실시간 게시글 탐색.

생산: 3단계 위저드(수단 선택 → 채널 선택 → 본문 작성)를 통한 빠른 글 등록.

소통: 댓글 또는 게시글 등록으로 전국의 대중교통 사용자들과의 교류.

만남: 대중교통 이용 수단이 또 다른 사람들과의 만남의 장으로 1:1 쪽지 기능(1,000 P)을 통해 인연을 찾는 공간으로 확대.

---

### 2.3 기본 브타(BMTA) 사용 플로우 (App Flow)

사용자는 브타(BMTA) 앱을 선택하여 접속한다

접속 시 디폴트인 브타 피드 메뉴에서 전체로 필터링된 모든 게시글을 한 눈으로 본다.

상단에는 핫이슈로 최근에 사람들이 많이 Boom Up(좋아요)이 높은 게시글이 있고 아래 영역에는 실시간으로 최근 등록된 게시글부터 보여진다.

글을 읽는 것은 비로그인도 가능하지만 글쓰기와 댓글을 쓰기위해 버튼을 누르면 로그인 여부를 판단한다.

로그인한 회원은 댓글이나 글쓰기가 가능하고 비회원인경우 로그인 화면으로 이동한다.

계정이 있는 경우 로그인을 하고, 없는 경우 회원가입을 진행한다.

브타 피드 메뉴의 게시글 상세페이지에서 Boom Up(좋아요) 개수와 달린 댓글을 확인하고 참여한다.

사용자 프로필을 선택하여 쪽지를 보내면서 개인적인 교류를 확대한다.

---

## 3. 기능적 요구사항 (Functional Requirements)

### 3.1 버스타 (Bus)

전국 버스 노선 실시간 위치 및 정류장별 도착 예정 시간(ETA) 제공.

즐겨찾기 기능을 통한 노선 퀵 액세스 지원.

---

### 3.2 지하철타 (Subway)

전국 5대 권역 지하철 노선도 및 역별 실시간 열차 위치 정보 제공.

역 선택 시 해당 역과 연결된 커뮤니티 피드 연동.

---

### 3.3 브타 피드 (BMTA Feed)

2단 필터 시스템: 1행(버스/지하철/택시) x 2행(전체/실시간 톡/사람을 찾아요/분실물 찾아요/이용 후기) 조합 필터링.

핫이슈 알고리즘: 조회수와 Boom Up(좋아요) 수치 기반 실시간 인기글 최상단 노출.

비 로그인 회원은 읽기만 가능

Boom Up(좋아요) 는 한 유저당 게시글에 1회 등록 가능하며 재등록 시 좋아요 취소

포인트 적립은 Boom Up이 유효하게 적용됐을때만 적립됨, 만약 누군구가 Boom Up(좋아요)를 눌렀다가 재클릭하여 취소할 경우 적립된 포인트는 다시 1만큼 차감

---

### 3.4 포인트 및 수익화 (Monetization)

포인트 적립: 내 글이 'Boom Up(좋아요)'을 받을 때마다 1P 적립.

쪽지 시스템: 1:1 소통을 위한 쪽지 발송 시 1,000P 차감 (스팸 방지 허들).

쪽지 1,000P 만큼 있는 경우 발송이 가능하고 발송 시점에 1,000P가 없는 경우 전송 불가

결제 연동: 외부 PG사를 통한 유료 포인트 충전 기능.

---

### 3.5 회원 정책

비로그인 회원 : 버스타/지하철타 메뉴 이용 가능, 게시글 읽기 가능

로그인 회원 : 버스타/지하철타 메뉴 이용 가능, 게시글 작성, 댓글 작성, 쪽지 보내기 가능

로그인 방식은 이메일 인증을 통한 회원 가입으로 인증된 이메일로 로그인 가능

닉네임은 초기 회원 가입 시 랜덤으로 생성되고 사용자가 원하는 경우 변경 가능함

---

## 4. 기술적 요구사항 (Technical Requirements)

### 4.1 개발 스택

프론트엔드: Flutter (iOS & Android 단일 코드베이스).

백엔드/인프라: Google Firebase (Firestore, Auth, Cloud Functions, Messaging).

외부 API: 공공데이터포털(국토교통부, 지자체 별 버스/지하철 실시간 정보 API).

---

### 4.2 성능 및 보안

실시간 데이터는 사용자 경험을 해치지 않는 범위에서 적절한 갱신 주기로 업데이트한다.

유료 기능 사용 시 휴대폰 본인확인(CI/DI 기반) 필수 적용.

주민등록번호 등 민감 개인정보 직접 수집 금지.

---

## 5. 데이터 개념정의 (Data Function)

### 5.1 버스타

정의 : 실시간 공공 API를 연동하여 버스 도착 정보를 시각화하고 커뮤니티로 연결하는 게이트웨이 역할을 합니다.

행동/규칙 : 

1. 실시간 위치 정보  
버스 실시간 위치 정보를 특정 시간 주기로 API 재호출 및 데이터 갱신 수행.

2. 거리순 정렬 로직  
GPS 권한 허용 시 사용자 현위치 기준 반경 500m 내 정류장 리스트 우선순위 노출.

3. 피드 숏컷(Deep-Link)  
도착 정보 카드의 ‘피드’ 아이콘 클릭 시, routeId를 파라미터로 전달하여 해당 노선 필터링된 [브타 피드] 로 이동

---

### 5.2 지하철타

정의 :  복잡한 지하철 네트워크를 직관적인 노선도로 제공하고, 열차의 실시간 위치를 동적으로 마킹합니다.

행동/규칙 : 

1. 실시간 위치 정보  
지하철 실시간 위치 정보를 특정 시간 주기로 API 재호출 및 데이터 갱신 수행.

2. 환승 정보 매핑  
특정 역 클릭 시 연동된 타 호선들의 도착 정보를 취합하여 통합 노출.

3. 역 검색  
특정 역 검색 시 도착 정보를 취합하여 노출하고 즐겨찾기 기능을 제공.

---

### 5.3 브타 피드 (BMTA Feed)

정의 : 사용자의 모든 이동 맥락을 담은 게시글을 필터로 분류하고 실시간으로 노출되는 브타의 핵심 서비스

행동/규칙 : 

1. 복합 필터링 (Dual-Layer)  
전체 게시글 페이지에서 1단 필터(교통 수단(Channel – 버스/지하철/택시)) 와  
2단 필터(카테고리(Topic – 전체/실시간 톡/사람을 찾아요/분실물 찾아요/이용 후기)) 로직으로 게시글 노출하며  
필터 변경 시 무한 스크롤 리스트 초기화 후 쿼리 재실행.

2. 핫이슈 랭킹 알고리즘  
조회수와 Boom Up(좋아요)을 가중치로 반영한 실시간 인기글 정렬  
구체적인 알고리즘은 구현 단계에서 별도 정의

3. 포인트 누적  
타 유저가 'Boom Up(좋아요)' 클릭 시, 작성자의 포인트를 즉시 1P 증가시켜서 기존 포인트에 실시간으로 합산

## 6. 로드맵 (Roadmap)

### 6.1 Phase 1: MVP 출시 (현재 단계)

버스/지하철 실시간 위치 정보 + 통합 커뮤니티 + 기본 포인트 시스템.

---

### 6.2 Phase 2: 기술 고도화

정밀 하차 알람: Wi-Fi BSSID 및 안내방송 인식 기반 하차 타이밍 알림.

---

### 6.3 Phase 3: 비즈니스 확장

태그리스 결제: BLE 센서 기반 원격 승하차 제어 및 자동 결제.

O2O 커머스: 지하철 역내 매장 스마트 오더 연동.

---

## 7. 작업 방식 (AI Rule)

### 7.1 작업 방식

한 번에 하나의 작업만 진행한다

작업 전 변경/생성 파일 목록을 먼저 제시한다

요청하지 않은 리팩토링은 하지 않는다

추측이 필요한 경우 반드시 질문한다

---

### 7.2 행동 제한 선언

이 PRD에 명시되지 않은 기능, 구조 변경, 최적화, 확장은 명시적으로 요청하지 않는 한 수행하지 않는다.

데이터 구조(엔티티, 관계)에 대한 변경은 명시적으로 요청하지 않는 한 제안하지 않는다.

---

### 7.3 작업 순서 (답변 방식)

변경/생성 파일 목록 공유

작업 범위(이번 작업에서 하는 것 / 안 하는 것) 설명

구현된 내용 설명

간단 테스트 시나리오(수동 기준)

현재까지 작업된 내역 체크리스트 작성
